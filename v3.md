# Arc v3 — Design Notes

Internal planning doc. Not shipped in the npm package.

## Why v3

v2 proved the resource-oriented model works. But real-world usage (especially AI agents building apps) exposed friction:

- `additionalRoutes` + `wrapHandler` is confusing — agents fall back to raw Fastify
- `schemaOptions.fieldRules` is stringly-typed, no autocomplete
- Permission helpers are powerful but verbose for common cases
- `toPlugin()` is invisible to AI agents
- QueryParser config duplicates between constructor and schemaOptions

v3 keeps `defineResource()` — same config object pattern, just cleaner fields.

---

## 1. Cleaner defineResource()

### Problem
v2 leaks internals (`_appliedPresets`, `wrapHandler`), duplicates config, and has unintuitive names.

### v3 Design

```typescript
import { defineResource, field, public, auth, roles, owner } from '@classytic/arc';

const product = defineResource({
  name: 'product',
  model: ProductModel,                          // auto-detects adapter (MongoKit/Prisma/custom)

  fields: {
    name: field.string().required(),
    price: field.number().min(0),
    category: field.enum('electronics', 'books', 'food'),
    internalScore: field.number().hidden(),
    createdAt: field.date().readonly(),
  },

  filter: ['category', 'status'],               // single source of truth (no more dual config)
  sort: ['createdAt', 'price'],
  operators: ['gte', 'lte', 'in', 'ne'],

  permissions: {
    list: public(),
    get: public(),
    create: roles('admin', 'editor'),
    update: owner('createdBy'),
    delete: roles('admin'),
  },

  presets: ['softDelete'],

  routes: [
    { method: 'POST', path: '/approve', handler: 'approve' },
    { method: 'POST', path: '/webhook', handler: webhookFn, raw: true },
    { method: 'GET', path: '/trending', handler: trendingFn, mcp: true },
  ],
});
```

**What changed from v2:**

| v2 | v3 | Why |
|----|-----|-----|
| `adapter: createMongooseAdapter(...)` | `model: ProductModel` | Auto-detect, zero boilerplate |
| `schemaOptions.fieldRules: { name: { type: 'string' } }` | `fields: { name: field.string() }` | Type-safe, autocomplete, no typos |
| `schemaOptions.filterableFields` + `QueryParser({ allowedFilterFields })` | `filter: [...]` | Single source of truth |
| `additionalRoutes: [{ wrapHandler: true }]` | `routes: [{ handler: 'methodName' }]` | Clear naming, no boolean confusion |
| `permissions: { list: allowPublic() }` | `permissions: { list: public() }` | Shorter, same function |

### What stays the same
- `defineResource()` returns `ResourceDefinition` (same internal type)
- Config object pattern (no builder)
- All v2 fields still accepted (deprecated, logged warning)
- Same BaseController, same pipeline, same adapters internally

## 2. routes Replaces additionalRoutes

### Problem
`additionalRoutes` + `wrapHandler: true/false` is the #1 confusion point. AI agents don't understand it and fall back to raw Fastify routes.

### v3 Design

```typescript
routes: [
  // String handler = controller method → full pipeline + MCP tool
  { method: 'POST', path: '/approve', handler: 'approve' },

  // Function handler = inline → full pipeline + MCP tool
  { method: 'GET', path: '/stats', handler: async (ctx) => ({ count: 42 }) },

  // raw: true = raw Fastify handler → no pipeline, no MCP
  { method: 'POST', path: '/webhook', handler: webhookFn, raw: true },

  // Explicit MCP config on action
  { method: 'POST', path: '/fulfill', handler: 'fulfill', mcp: {
    description: 'Fulfill this order',
    annotations: { destructiveHint: true },
  }},
]
```

**Rules:**
- `handler: 'string'` → calls controller method by name → goes through BaseController pipeline
- `handler: function` → inline handler → goes through pipeline
- `raw: true` → raw Fastify, escapes everything, no MCP
- Default: pipeline + MCP tool auto-generated. `mcp: false` to skip MCP.

## 3. Unified Query Config

### Problem
Same fields configured in two places:
```typescript
// v2 — duplicated
queryParser: new QueryParser({ allowedFilterFields: ['status', 'category'] }),
schemaOptions: { filterableFields: ['status', 'category'] },
```

### v3 Design

```typescript
// v3 — single source
filter: ['status', 'category'],
sort: ['createdAt', '-price'],
operators: ['gte', 'lte', 'in', 'ne'],
```

Arc creates the QueryParser internally from these arrays. Users who need a custom parser can still pass one:
```typescript
queryParser: new CustomParser({ ... }),  // overrides filter/sort/operators
```

## 4. Field Definitions — Type-Safe

### Problem
`fieldRules` is `Record<string, { type: string, required?: boolean }>` — no autocomplete, easy to typo `'stirng'`.

### v3 Design

```typescript
import { field } from '@classytic/arc';

fields: {
  name: field.string().required().description('Product name'),
  price: field.number().min(0).max(99999),
  category: field.enum('electronics', 'books', 'food').default('other'),
  tags: field.array(field.string()),
  metadata: field.object({ color: field.string() }),
  internalScore: field.number().hidden(),
  createdAt: field.date().readonly().systemManaged(),
}
```

**Benefits:**
- Full IDE autocomplete
- Type inference for create/update body types
- MCP schemas, OpenAPI schemas, Zod schemas all derived from same definition
- `.hidden()` = not in API or MCP, `.readonly()` = not in create/update, `.systemManaged()` = both

**Internal:** `field.string()` returns a plain object (`{ type: 'string', ... }`) — same `FieldRuleEntry` as v2. It's just a helper, not a class.

## 5. Auto-Detect Adapter

### Problem
```typescript
// v2 — three imports, two instantiations
import { createMongooseAdapter } from '@classytic/arc/adapters';
import { Repository } from '@classytic/mongokit';
adapter: createMongooseAdapter({ model: ProductModel, repository: new Repository(ProductModel) }),
```

### v3 Design
```typescript
// Mongoose model → auto-detect
model: ProductModel,

// Prisma delegate → auto-detect
model: prisma.product,

// Custom repository → direct
repository: customRepository,
```

Detection:
- Has `.schema` + `.modelName` → Mongoose → auto-creates Repository + adapter
- Has `.findMany` + `.create` → Prisma → auto-creates PrismaAdapter
- Implements `RepositoryLike` → use directly

## 6. Registration — No More toPlugin()

### Problem
AI agents don't know `resource.toPlugin()` exists. They write `fastify.get(...)` instead.

### v3 Design
```typescript
const app = await createApp({
  resources: [productResource, orderResource],
  mcp: { auth: false },
});
```

No `toPlugin()`, no `app.register()` per resource. `createApp` handles registration.

For gradual migration, `app.register(resource.toPlugin())` still works.

---

## 7. Permission Shorthand

### Problem
```typescript
// v2 — verbose
permissions: {
  list: allowPublic(),
  get: allowPublic(),
  create: requireAuth(),
  update: requireOwnership('createdBy'),
  delete: requireRoles('admin'),
}
```

### v3 Design
```typescript
// Short helpers (same functions, shorter names)
permissions: {
  list: public(),
  get: public(),
  create: auth(),
  update: owner('createdBy'),
  delete: roles('admin'),
}

// Preset strings for common patterns
permissions: 'publicRead, authWrite',    // list+get public, CUD require auth
permissions: 'ownerCrud',               // owner can CRUD their own
permissions: 'adminOnly',               // all ops require admin role
```

Custom functions still work — presets are just shorthand:
```typescript
permissions: {
  list: (ctx) => ({ granted: true, filters: { orgId: ctx.user.orgId } }),
}
```

## 8. App-Level MCP Config

### Problem
MCP is registered separately. AI agents forget it.

### v3 Design
```typescript
const app = await createApp({
  resources: [product, order],
  mcp: { auth: false },              // auto-generate from all resources
  // mcp: false,                     // disabled (default)
  // mcp: true,                      // enabled, no auth
});
```

Per-resource MCP control via `routes` with `mcp: false` or resource-level `mcp: false`.

## 10. Migration

| v2 | v3 | Breaking? |
|----|-----|-----------|
| `additionalRoutes` | `routes` | No — v2 field still works (deprecated) |
| `wrapHandler: true` | `handler: 'methodName'` | No — v2 field still works |
| `wrapHandler: false` | `raw: true` | No — v2 field still works |
| `resource.toPlugin()` | `createApp({ resources })` | No — both work |
| `createMongooseAdapter()` | `model: Model` | No — both work |
| `schemaOptions.fieldRules` | `fields: { ... }` | No — both work |
| `filterableFields` + `QueryParser` | `filter: [...]` | No — both work |
| `allowPublic()` | `public()` | No — both exported |

**Zero breaking changes.** All v2 config fields accepted with deprecation warnings. Codemod for automated migration: `npx @classytic/arc migrate v2-to-v3`

## 11. Deferred from v2.5 (Candidate for v3)

### wrapHandler default flip
`wrapHandler: false` is used 100% of the time in real apps. In v3 (`routes`), string handler = pipeline, function handler = pipeline, `raw: true` = escape. The boolean is gone entirely.

### additionalRoutes hook integration
v2: `wrapHandler: false` routes bypass the hook pipeline by design. v3: all `routes` (except `raw: true`) go through hooks — the `routes` redesign fixes this naturally since non-raw routes always go through the controller pipeline.

### bulk preset validation
Fail at registration time if the repository doesn't have `createMany`/`updateMany`/`deleteMany` instead of failing at request time.

### MCP SDK v2 migration
Package split: `@modelcontextprotocol/sdk` → `@modelcontextprotocol/server` + `@modelcontextprotocol/node`. Transport rename: `StreamableHTTPServerTransport` → `NodeStreamableHTTPServerTransport`. Schema: raw shapes → `z.object()` required. Handler context: `extra` → `ctx.mcpReq`. See `src/integrations/mcp/MIGRATION-V2.md` for exact diffs — 5 source files, mechanical changes.

### Node.js 22+ only
Drop Node 20. Align with MongoKit 3.4.3 requirement.

## 12. Team Feedback (arc-ai production usage — April 2026)

From the arc-ai team running a production Vercel-style AI chatbot on Arc + MongoKit:

### What scored 9/10
- `defineResource` + `idField` + `additionalRoutes` — one config object
- `cascadePlugin` on MongoKit — declarative, fires for all delete paths
- `streamResponse: true` — SSE without boilerplate
- `request.signal` passthrough — abort propagation works
- Presets (bulk, softDelete, ownedByUser, audited) — composable
- `createApp({ auth: false })` — zero-config dev mode
- Hook system at 4 levels (inline, preset, app-level, wildcard)
- Events auto-published on CRUD

### What cost time (fix in v3)
- `wrapHandler: false` on every custom route → fixed by `routes` redesign (§2)
- No docs/examples for AI SDK streaming → add cookbook examples
- `SchemaOptions.filterableFields` not in TypeScript interface → fix type definition
- Redundant QueryParser + filterableFields config → fixed by `filter` array (§3)

### Improvements shipped in v2.5.x (don't defer to v3)
- `include` option on mcpPlugin (alongside existing `exclude`)
- Per-resource tool naming: `overrides.job.names.get = 'get_job_by_id'`
- Per-resource tool prefix: `overrides.job.toolNamePrefix = 'db'`
- Auto-derive `filterableFields` from `QueryParser.allowedFilterFields`
- Auto-derive `allowedOperators` from `QueryParser.allowedOperators`
- Operator-suffixed filter fields in MCP schemas (`price_gt`, `price_lte`)
- `GET /mcp/health` diagnostic endpoint
- Auth failure WARN logging (silent 401 debugging)
- Auth cache for stateless mode (avoids redundant DB calls)
- `createTestMcpClient()` test utility
- `mcpHandler` on additionalRoutes (MCP without `wrapHandler`)

### Improvements shipped in v2.5.2 (don't defer to v3)
- `roles()` unified permission helper — checks platform + org roles
- `requireRoles({ includeOrgRoles: true })` backward-compatible option
- `AdditionalRoute.handler` accepts `ControllerHandler` (no more `as any`)
- Bracket notation filters work (`name[contains]`, `price[gte]`) — AJV type stripping
- `excludeFields` / `readonlyFields` / `immutable` respected by Mongoose schema gen
- BodySanitizer strips immutable fields on update
- `createApp({ resources })` — no `toPlugin()` needed
- `loadResources(dir)` — auto-discover `*.resource.ts` from directory
- Audit `extractUserId` DB-agnostic (Mongoose ObjectId, string, number)
- Response schema `additionalProperties: true` — no field stripping
- Mongoose type mapping: Array elements, Mixed, Map, Buffer, Decimal128, UUID
- `SchemaOptions.filterableFields` added to TypeScript interface (was missing)

### What to add in v3
- **Cookbook examples:** AI streaming, file uploads, webhook receivers
- **AI SDK streaming docs:** Document the `streamResponse + createUIMessageStream` pattern
- **`routes` with default MCP:** All non-raw routes auto-generate MCP tools
- **Frontend SSE utilities:** `arc-next` lacks streaming/SSE hooks
- **`resourceDir` on createApp** — `createApp({ resourceDir: './src/resources' })` as sugar over `loadResources`
- **Permission preset strings** — `permissions: 'publicRead, authWrite'` alongside function-based

### What's resolved (was in v3 plan, shipped in v2.5.x instead)
- ~~`SchemaOptions` type fix~~ → shipped in v2.5.2
- ~~Redundant QueryParser + filterableFields~~ → auto-derived from QueryParser in v2.5.0
- ~~BA org role mismatch with `requireRoles()`~~ → `roles()` helper in v2.5.2
- ~~`AdditionalRoute.handler` type gap~~ → `ControllerHandler` in union in v2.5.2
- ~~Bracket notation query rejections~~ → AJV type stripping in v2.5.2
- ~~Manual `toPlugin()` registration~~ → `createApp({ resources })` in v2.5.2

## Non-Goals

- No workflow engine — use Streamline/Temporal
- No email/notification — app-level
- No bundled peer deps — tree-shakable
- No CJS — ESM-only
- No GraphQL — REST + MCP
- No builder pattern — config objects are simpler and don't break
- No full-stack — backend only

---

## Timeline

No deadline. Ship when field helpers are solid and v2 compat is fully tested.
