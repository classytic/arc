# Org Module

Multi-tenant organization scoping for SaaS applications. Covers everything
from single-org isolation to multi-dimension tenancy (org + branch + project),
parent-child organization hierarchies, and mixed human + API-key authorization
on the same routes.

## Quick Start

```typescript
import { createApp } from '@classytic/arc/factory';
import { createBetterAuthAdapter } from '@classytic/arc/auth';
import { defineResource } from '@classytic/arc';
import {
  allOf, anyOf,
  requireOrgMembership, requireOrgRole,
  requireServiceScope,    // OAuth-style scope strings for API keys
  requireScopeContext,    // app-defined dimensions (branch, project, region)
  requireOrgInScope,      // parent-child org hierarchies
} from '@classytic/arc/permissions';

const app = await createApp({
  preset: 'production',
  auth: {
    type: 'betterAuth',
    betterAuth: createBetterAuthAdapter({
      auth: getAuth(),
      orgContext: true, // Auto-resolve org membership into request.scope
    }),
  },
});

const orderResource = defineResource({
  name: 'order',
  presets: ['multiTenant'],
  permissions: {
    // Any org-bound caller — member, service (API key), or elevated admin
    list: requireOrgMembership(),

    // Human admins OR API keys with the right OAuth scope
    create: anyOf(
      requireOrgRole('admin', 'editor'),
      requireServiceScope('orders:write'),
    ),

    // Strictly humans
    delete: requireOrgRole('admin'),
  },
});

await app.register(orderResource.toPlugin());
```

## Request Scope

Every request has a `request.scope` field — a discriminated union that
replaces scattered org context fields. **Five kinds**, all opt-in:

```typescript
import type { RequestScope } from '@classytic/arc/scope';

type RequestScope =
  | { kind: 'public' }
  | { kind: 'authenticated'; userId?: string; userRoles?: string[] }
  | {
      kind: 'member';
      userId?: string;
      userRoles: string[];
      organizationId: string;
      orgRoles: string[];
      teamId?: string;
      /** App-defined dimensions: branch, project, region, workspace, … */
      context?: Readonly<Record<string, string>>;
      /** Parent orgs (closest-first) for hierarchy checks */
      ancestorOrgIds?: readonly string[];
    }
  | {
      kind: 'service';
      clientId: string;
      organizationId: string;
      /** OAuth-style scope strings, e.g. ['orders:read', 'orders:write'] */
      scopes?: readonly string[];
      context?: Readonly<Record<string, string>>;
      ancestorOrgIds?: readonly string[];
    }
  | {
      kind: 'elevated';
      userId?: string;
      organizationId?: string;
      elevatedBy: string;
      context?: Readonly<Record<string, string>>;
      ancestorOrgIds?: readonly string[];
    };
```

| Kind | Identity | Org context | Set by |
|------|----------|-------------|--------|
| `public` | none | none | Default for anonymous requests |
| `authenticated` | userId, userRoles | none | Logged in but no active org selected |
| `member` | userId, userRoles | organizationId + orgRoles (+ teamId, context, ancestorOrgIds) | Better Auth org plugin / JWT custom auth |
| `service` | clientId, scopes | organizationId (required) | API-key auth via `PermissionResult.scope` |
| `elevated` | userId | organizationId optional | Platform admin via `x-arc-scope: platform` header + elevation plugin |

### Type Guards & Accessors

```typescript
import {
  // Type guards
  isMember, isService, isElevated, isAuthenticated, hasOrgAccess,
  // Identity / org accessors
  getUserId, getUserRoles, getOrgId, getOrgRoles, getTeamId, getClientId,
  // Service scopes
  getServiceScopes,
  // App-defined dimensions
  getScopeContext, getScopeContextMap,
  // Parent-child hierarchy
  getAncestorOrgIds, isOrgInScope,
} from '@classytic/arc/scope';

const scope = request.scope;

if (isElevated(scope)) {
  // Platform admin — typically full access
}

if (hasOrgAccess(scope)) {
  // member, service, or elevated — anything that carries an organizationId
  const orgId = getOrgId(scope);
}

if (isService(scope)) {
  const clientId = getClientId(scope);          // string
  const grantedScopes = getServiceScopes(scope); // readonly string[]
}

// Custom dimensions populated by your auth function
const branchId = getScopeContext(scope, 'branchId');
const allDimensions = getScopeContextMap(scope); // Readonly<Record<string,string>> | undefined

// Parent-child checks
const ancestors = getAncestorOrgIds(scope);     // readonly string[] (always)
if (isOrgInScope(scope, 'acme-holding')) {
  // current org is acme-holding OR acme-holding appears in ancestorOrgIds
}
```

### Scope in Controllers

Controllers receive scope via `req.metadata._scope`:

```typescript
import { getOrgId } from '@classytic/arc/scope';
import { getControllerScope } from '@classytic/arc/core';

class OrderController extends BaseController {
  async getStats(req: IRequestContext): Promise<IControllerResponse> {
    const scope = getControllerScope(req);

    // Accept member, service, or elevated — anything with org context
    if (!hasOrgAccess(scope)) {
      return { success: false, error: 'Org context required', status: 403 };
    }

    const stats = await Order.aggregate([
      { $match: { organizationId: getOrgId(scope) } },
      { $group: { _id: '$status', count: { $sum: 1 } } },
    ]);

    return { success: true, data: stats };
  }
}
```

## How Scope Gets Set

### Better Auth Adapter

When `orgContext: true`, the adapter:
1. Resolves the user's session
2. Looks up active organization membership (and team, if the BA `teams` plugin is enabled)
3. Sets `request.scope` based on the result:

```
No session                       → { kind: 'public' }
Session, no active org           → { kind: 'authenticated' }
Session + org membership         → { kind: 'member', organizationId, orgRoles, teamId? }
```

The Better Auth adapter does NOT populate `context` or `ancestorOrgIds` —
those are app-defined and you set them yourself in a downstream auth hook
(see below).

### JWT Auth

For JWT apps, scope defaults to `authenticated` for logged-in users. Use the
`resolveOrgFromHeader` utility for org header resolution:

```typescript
import { resolveOrgFromHeader } from '@classytic/arc/scope';

app.addHook('onRequest', resolveOrgFromHeader({
  resolveMembership: async (userId, orgId) => {
    const member = await MemberModel.findOne({ userId, orgId });
    return member ? { roles: member.roles.split(',') } : null;
  },
}));
```

### API Key Auth (service scope)

Custom API-key authentication ships scope through the permission layer using
`PermissionResult.scope` — no separate auth plugin required:

```typescript
import type { PermissionCheck } from '@classytic/arc/permissions';

export function requireApiKey(): PermissionCheck {
  return async ({ request }) => {
    const apiKey = request.headers['x-api-key'] as string | undefined;
    if (!apiKey) return { granted: false, reason: 'Missing API key' };

    const client = await ClientModel.findOne({ apiKey });
    if (!client) return { granted: false, reason: 'Invalid API key' };

    return {
      granted: true,
      // Install service scope — Arc writes this onto request.scope and
      // the multiTenantPreset filter picks it up automatically.
      scope: {
        kind: 'service',
        clientId: String(client._id),
        organizationId: String(client.companyId),
        scopes: client.allowedScopes,  // ['orders:read', 'orders:write']
      },
    };
  };
}

// Then compose with org membership and OAuth-style scope checks
permissions: {
  list:   allOf(requireApiKey(), requireOrgMembership(), requireServiceScope('orders:read')),
  create: allOf(requireApiKey(), requireOrgMembership(), requireServiceScope('orders:write')),
}
```

### Adding Custom Dimensions and Ancestors

For routes that need branch/project/region scoping or parent-child
hierarchy, populate `context` and `ancestorOrgIds` in your own auth hook
(arc takes no position on the source — read from JWT claims, request
headers, or your own DB):

```typescript
app.addHook('onRequest', async (request) => {
  if (!request.scope || request.scope.kind !== 'member') return;

  // Read from headers, claims, or your own data model
  const branchId = request.headers['x-branch-id'] as string | undefined;
  const projectId = request.headers['x-project-id'] as string | undefined;
  const ancestors = await orgRepo.findAncestors(request.scope.organizationId);

  // Replace scope with the enriched version (cast through unknown to bypass
  // the readonly contract on context — your hook owns scope construction)
  (request as { scope: typeof request.scope }).scope = {
    ...request.scope,
    context: { branchId, projectId } as Readonly<Record<string, string>>,
    ancestorOrgIds: ancestors.map(a => a.id),
  };
});
```

## Automatic Data Filtering

With the `multiTenant` preset, queries are automatically filtered by tenant
fields. The preset supports two modes — single-field for the common case,
multi-field for branch/project/region scoping in lockstep.

### Single-field (default)

```typescript
import { multiTenantPreset } from '@classytic/arc/presets';

multiTenantPreset({ tenantField: 'organizationId' })

// GET /orders (caller in org-123)
// → repository.findAll({ filter: { organizationId: 'org-123' } })

// POST /orders (caller in org-123)
// → request.body.organizationId = 'org-123'
```

Works identically for `member`, `service`, and `elevated` callers — all
three carry an `organizationId` and the preset filters by it via
`hasOrgAccess()`.

### Multi-field (branch / project / region / team)

```typescript
multiTenantPreset({
  tenantFields: [
    { field: 'organizationId', type: 'org' },              // → getOrgId(scope)
    { field: 'teamId',         type: 'team' },             // → getTeamId(scope)
    { field: 'branchId',       contextKey: 'branchId' },   // → getScopeContext(scope, 'branchId')
    { field: 'projectId',      contextKey: 'projectId' },
  ],
})
```

Each entry is a `TenantFieldSpec`:
- `{ field, type: 'org' }` — reads `getOrgId(scope)`
- `{ field, type: 'team' }` — reads `getTeamId(scope)`
- `{ field, contextKey: '...' }` — reads `getScopeContext(scope, contextKey)`

**Fail-closed semantics**: if any required dimension is missing from the
caller's scope, list/get/update/delete return 403 with the specific missing
field name in the error message; create is rejected. Elevated scopes apply
whatever resolves and skip the rest (cross-context admin bypass).

`tenantField` and `tenantFields` are mutually exclusive — passing both
throws at preset construction.

### Elevated and service scope behavior

| Scope kind | Filter behavior |
|---|---|
| `member` | Filter by `scope.organizationId` (+ all configured dimensions) |
| `service` | Same as member — API key with org context gets the same isolation |
| `elevated` + `organizationId` | Filter by that org (admin viewing one org) |
| `elevated` + no `organizationId` | No filter (admin viewing all orgs cross-tenant) |
| `public` | 401 (strict) or fall-through (allowPublic mode) |
| `authenticated` (no org) | 403 |

## Elevation Plugin (Platform Admin)

For apps with platform admins who need cross-org access:

```typescript
import { createApp } from '@classytic/arc/factory';

const app = await createApp({
  auth: { type: 'betterAuth', betterAuth: adapter },
  elevation: {
    platformRoles: ['superadmin'],
    onElevation: (event) => auditLog.write({
      action: 'platform_elevation',
      userId: event.userId,
      targetOrg: event.organizationId,
    }),
  },
});
```

How it works:
- Without `x-arc-scope: platform` header, a superadmin is a regular user
- With `x-arc-scope: platform` + valid platform role → `elevated` scope
- With `x-arc-scope: platform` + wrong role → 403

```bash
# Regular request (superadmin as normal org member)
curl -H "Authorization: Bearer <token>" /orders
# scope = { kind: 'member', organizationId: '...', orgRoles: ['admin'] }

# Elevated request (platform admin mode)
curl -H "Authorization: Bearer <token>" \
     -H "x-arc-scope: platform" \
     -H "x-organization-id: target-org-id" /orders
# scope = { kind: 'elevated', organizationId: 'target-org-id', elevatedBy: 'user-id' }
```

## Guards

For route-level org enforcement:

// Require org context
fastify.get('/orders', {
  preHandler: [fastify.authenticate, requireOrg()],
  handler: listOrders,

// Require specific org role

  preHandler: [fastify.authenticate, requireOrgRole('manager')],

## Permission Functions

```typescript
import {
  // Org membership / roles
  requireOrgMembership,
  requireOrgRole,
  requireTeamMembership,
  // OAuth-style API key scopes
  requireServiceScope,
  // App-defined scope dimensions (branch, project, region, …)
  requireScopeContext,
  // Parent-child org hierarchy
  requireOrgInScope,
  // Static role × resource × action matrix
  createOrgPermissions,
  // Combinators
  allOf, anyOf, when,
} from '@classytic/arc/permissions';

// Any org-bound caller — member, service, or elevated
requireOrgMembership()

// Specific org role(s) — ANY-match logic. HUMAN-ONLY: services explicitly
// fail this check with a guidance reason pointing at requireServiceScope.
// For mixed routes that should accept both, compose with anyOf().
requireOrgRole('admin', 'editor')

// Active team membership (BA org plugin teams enabled)
requireTeamMembership()

// OAuth-style scope check on a service identity (API key). Variadic + array
// forms. Elevated bypass. Throws at construction if no scopes provided.
requireServiceScope('orders:write')
requireServiceScope('orders:read', 'orders:write')
requireServiceScope(['orders:read', 'orders:write'])

// App-defined scope dimensions — three call shapes
requireScopeContext('branchId')                                  // presence-only
requireScopeContext('region', 'eu')                              // exact value match
requireScopeContext({ branchId: 'eng-paris', projectId: 'p-1' }) // multi-key AND

// Parent-child org hierarchy. Static or dynamic target.
requireOrgInScope('acme-holding')                                // static
requireOrgInScope((ctx) => ctx.request.params.orgId)             // dynamic from route param
requireOrgInScope((ctx) => ctx.request.body?.organizationId)     // dynamic from body

// Static permission matrix per role
const perms = createOrgPermissions({
  statements: {
    invoice: ['create', 'update', 'delete', 'approve'],
    order: ['create', 'update'],
  },
  roles: {
    admin: { invoice: ['create', 'update', 'delete', 'approve'], order: ['create', 'update'] },
    accountant: { invoice: ['create', 'update', 'approve'], order: ['create'] },
    viewer: { invoice: [], order: [] },
  },
});

perms.can({ invoice: ['approve'] })  // PermissionCheck function
```

### Behavior summary

| Helper | Member | Service (API key) | Elevated | Notes |
|---|---|---|---|---|
| `requireOrgMembership()` | ✅ | ✅ | ✅ | Any org-bound caller |
| `requireOrgRole(roles)` | If role matches | ❌ deny with guidance | ✅ bypass | Human-only by design |
| `requireServiceScope(scopes)` | ❌ | If scope matches | ✅ bypass | OAuth-style, machine-only |
| `requireScopeContext(...)` | If keys match | If keys match | ✅ bypass | App-defined dimensions |
| `requireTeamMembership()` | If `teamId` set | (n/a — services don't have teams) | ✅ bypass | BA org plugin teams |
| `requireOrgInScope(target)` | If target = current or in `ancestorOrgIds` | If target = current or in `ancestorOrgIds` | ✅ bypass | Pure data check + elevated bypass |

### Mixed routes (human + machine)

When a route should accept both an org admin and an API key with the right
OAuth scope, compose explicitly with `anyOf`:

```typescript
permissions: {
  bulkImport: anyOf(
    requireOrgRole('admin'),                    // human path
    requireServiceScope('orders:bulk-write'),   // machine path
  ),
}
```

This is intentional rather than implicit. "API key bypasses role checks" is
the kind of footgun that ships data breaches — services must opt into
specific scope strings the same way OAuth clients do.

### Real-world scenario quick reference

| Scenario | Helper combination |
|---|---|
| Multi-tenant SaaS, single-org isolation | `multiTenantPreset({ tenantField: 'organizationId' })` + `requireOrgMembership()` |
| Single org with multiple branches | `multiTenantPreset({ tenantFields: [{field:'organizationId',type:'org'}, {field:'branchId',contextKey:'branchId'}] })` + `requireScopeContext('branchId')` |
| Multi-team within an org | `multiTenantPreset({ tenantFields: [..., {field:'teamId',type:'team'}] })` + `requireTeamMembership()` |
| API key with org isolation | `allOf(requireApiKey(), requireOrgMembership())` + multi-tenant preset |
| API key with explicit OAuth scopes | `allOf(requireApiKey(), requireServiceScope('jobs:write'))` |
| Holding company → subsidiaries (parent admin can act on children) | `requireOrgInScope((ctx) => ctx.request.params.orgId)` + populate `scope.ancestorOrgIds` from your auth hook |
| MSP managing many tenant orgs | Same — load all managed tenant ids into `ancestorOrgIds` |
| Region-locked endpoint (data residency) | `requireScopeContext('region', 'eu')` |
| Cross-tenant platform admin | Elevation plugin + `x-arc-scope: platform` header — `requireOrgRole`, `requireServiceScope`, `requireScopeContext`, `requireOrgInScope` all bypass automatically |

## Best Practices

1. **Use `request.scope` type guards** — `isMember()`, `isElevated()`, `getOrgId()` instead of raw field access
2. **Let presets handle filtering** — Use `multiTenant` preset instead of manual org filtering
3. **Use permission functions** — `requireOrgRole()` instead of manual role checks
4. **Elevation is opt-in** — Only enable elevation plugin if you need platform admin cross-org access
5. **Test all scope kinds** — Verify behavior for public, authenticated, member, and elevated
