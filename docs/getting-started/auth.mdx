# Authentication

Auth uses a **discriminated union** — pick a `type`:

```typescript
// Arc JWT
auth: { type: 'jwt', jwt: { secret, expiresIn: '15m', refreshSecret, refreshExpiresIn: '7d' } }

// Better Auth (recommended for SaaS with orgs)
import { createBetterAuthAdapter } from '@classytic/arc/auth';
auth: { type: 'betterAuth', betterAuth: createBetterAuthAdapter({ auth, orgContext: true }) }

// Custom Fastify plugin (must decorate fastify.authenticate)
auth: { type: 'custom', plugin: myAuthPlugin }

// Custom function (decorates fastify.authenticate directly)
auth: { type: 'authenticator', authenticate: async (req, reply) => { ... } }

// Disabled
auth: false
```

## Quick Start (JWT)

```bash
npm install @fastify/jwt@^10.0.0
```

```typescript
import { createApp } from '@classytic/arc/factory';

const app = await createApp({
  preset: 'production',
  auth: {
    type: 'jwt',
    jwt: {
      secret: process.env.JWT_SECRET,  // Required, 32+ chars
      expiresIn: '15m',
    },
  },
});
```

> Arc uses `@fastify/jwt` v10 (fast-jwt internally). If you only use `secret` + `expiresIn`, no migration needed from v9.

## Decorators

| Decorator | Description |
|-----------|-------------|
| `fastify.authenticate` | Verify JWT/session, set `request.user`. Returns 401 on failure. |
| `fastify.optionalAuthenticate` | Parse token if present, skip if absent. Never 401. |
| `fastify.authorize(...roles)` | Check `user.role`. `authorize('*')` = any auth user. |

```typescript
fastify.get('/protected', {
  preHandler: [fastify.authenticate],
  handler: async (request) => ({ user: request.user }),
});

fastify.post('/admin', {
  preHandler: [fastify.authenticate, fastify.authorize('admin')],
  handler: async (request) => { /* admins only */ },
});
```

## Token Lifecycle (JWT)

```typescript
// Issue tokens
const tokens = app.auth.issueTokens({ _id: user._id, email, roles });
// Returns: { accessToken, refreshToken?, expiresIn, tokenType: 'Bearer' }

// Verify refresh token (rejects access tokens)
const decoded = app.auth.verifyRefreshToken(refreshToken);
```

## Permissions

Use permission functions (not string arrays):

```typescript
import { defineResource, allowPublic, requireAuth, requireRoles } from '@classytic/arc';

defineResource({
  name: 'product',
  permissions: {
    list: allowPublic(),
    get: allowPublic(),
    create: requireRoles(['admin', 'editor']),
    update: requireRoles(['admin']),
    delete: requireRoles(['admin']),
  },
});
```

| Function | Description |
|----------|-------------|
| `allowPublic()` | No authentication required |
| `requireAuth()` | Any authenticated user |
| `requireRoles('admin', 'editor')` | At least one role matches — checks BOTH platform and org roles by default. Variadic or array form. |
| `requireRoles(['admin'], { includeOrgRoles: false })` | Platform-only opt-out |
| `requireOrgMembership()` | Any org-bound caller (member, service, or elevated) |
| `requireOrgRole('admin')` | Org membership role — humans only; services use `requireServiceScope` |
| `requireServiceScope('orders:write')` | OAuth-style scope check on `service` (API key) identity |
| `requireScopeContext('branchId')` | App-defined scope dimensions (branch, project, region, …) — see [Permissions](./permissions#requirescopecontext) for the 3 call shapes |
| `requireOrgInScope((ctx) => ctx.request.params.orgId)` | Parent-child org hierarchy check (caller's current org or any ancestor) |
| `requireTeamMembership()` | Active team in current org |
| `requireOwnership('userId')` | Resource owner only (elevated bypasses) |
| `denyAll(reason?)` | Always deny |
| `allOf(p1, p2)` | AND — all must pass; scope and filters propagate between children |
| `anyOf(p1, p2)` | OR — any can pass |

> **Mixed human + machine routes**: `requireOrgRole` denies service scopes
> by design (with a guidance reason). For routes that should accept both
> an org admin AND an API key with the right OAuth scope, compose with
> `anyOf`:
>
> ```typescript
> permissions: {
>   create: anyOf(
>     requireOrgRole('admin'),                       // human path
>     requireServiceScope('orders:write'),           // machine path
>   ),
> }
> ```

**Custom permission:**

```typescript
import type { PermissionCheck } from '@classytic/arc/permissions';

const requirePremium = (): PermissionCheck => async (ctx) => {
  if (!ctx.user) return { granted: false, reason: 'Unauthorized' };
  return { granted: ctx.user.isPremium };
};
```

See [Permissions](./permissions) for full reference.

## Better Auth (SaaS)

```typescript
import { createBetterAuthAdapter } from '@classytic/arc/auth';

const adapter = createBetterAuthAdapter({
  auth: getAuth(),         // Your betterAuth() instance
  basePath: '/api/auth',
  orgContext: true,        // Auto-resolve org membership into request.scope
});

const app = await createApp({
  auth: { type: 'betterAuth', betterAuth: adapter },
});
```

**Org context flow** (when `orgContext: true`):
1. Gets session via Better Auth
2. Reads `session.activeOrganizationId`, or falls back to `x-organization-id` header (for API key auth)
3. Looks up org membership
4. Splits roles: `"admin,recruiter"` → `['admin', 'recruiter']`
5. Sets `request.scope`: `{ kind: 'member', organizationId, orgRoles: string[] }`

See [Org Module](./org) for multi-tenant setup.

### Mongoose Populate with Better Auth Collections

Better Auth uses the raw MongoDB driver — not Mongoose. If your resource
schemas reference BA collections (e.g. `ref: 'user'`, `ref: 'organization'`),
Mongoose `populate()` throws `MissingSchemaError` because no Mongoose model
exists for those collections.

**Fix:** Use the optional helper at the dedicated subpath
`@classytic/arc/auth/mongoose`. It registers `strict: false` stub Mongoose
models for the BA collections you're using, with full plugin coverage and
auto-handling of `usePlural` / custom model names. Lives behind a subpath
export so non-Mongoose users (Prisma/Drizzle/Kysely) never get Mongoose
pulled into their bundle.

```typescript
import mongoose from 'mongoose';
import { betterAuth } from 'better-auth';
import { mongodbAdapter } from '@better-auth/mongo-adapter';
import { organization } from 'better-auth/plugins';
import { registerBetterAuthMongooseModels } from '@classytic/arc/auth/mongoose';

const auth = betterAuth({
  database: mongodbAdapter(mongoose.connection.getClient().db()),
  plugins: [organization({ teams: { enabled: true } })],
  // ...
});

// Run AFTER betterAuth() and BEFORE any populate() calls.
// Default is core only (`user`, `session`, `account`, `verification`) —
// every plugin set is opt-in.
registerBetterAuthMongooseModels(mongoose, {
  plugins: ['organization', 'organization-teams'],
  // For separate @better-auth/* packages (passkey, sso, api-key):
  extraCollections: ['passkey', 'ssoProvider'],
});
```

Plugin coverage (researched against BA 1.6 docs, all opt-in):

| Plugin key | Collections registered |
|---|---|
| `core` (always) | `user`, `session`, `account`, `verification` |
| `organization` | `organization`, `member`, `invitation` |
| `organization-teams` | `team`, `teamMember` |
| `twoFactor` | `twoFactor` |
| `jwt` | `jwks` |
| `oidcProvider` / `oauthProvider` (alias) | `oauthApplication`, `oauthAccessToken`, `oauthConsent` |
| `mcp` | (reuses oidcProvider schema per BA docs) |
| `deviceAuthorization` | `deviceCode` |

Field-only plugins (admin, username, phoneNumber, magicLink, emailOtp,
anonymous, bearer, multiSession, siwe, lastLoginMethod, genericOAuth)
need NO entry — `strict: false` stubs round-trip extra fields automatically.

**Idempotent + de-duplicating** — safe to call repeatedly under HMR;
overlapping plugin sets won't crash with `OverwriteModelError`.

> **Note:** `arc init --better-auth --mongokit` includes the call automatically.

### API Key Auth

When using Better Auth's `apiKey()` plugin, API keys create synthetic sessions that have **no org context**. Arc bridges this gap with the `x-organization-id` header:

```
x-api-key: ak_live_...           # Authentication
x-organization-id: org_abc123    # Org context for tenant-scoped resources
```

Arc's adapter auto-detects the `apiKey()` plugin and generates matching OpenAPI security schemes. See [OpenAPI docs](../production-ops/openapi#external-paths--dynamic-security) for spec details.

## Custom Auth

### Custom Plugin

```typescript
const app = await createApp({
  auth: { type: 'custom', plugin: myAuthPlugin },
  // Plugin must decorate fastify.authenticate
});
```

### Custom Function

```typescript
const app = await createApp({
  auth: {
    type: 'authenticator',
    authenticate: async (req, reply) => {
      const session = await validateSession(req);
      if (!session) reply.code(401).send({ error: 'Unauthorized' });
      req.user = session.user;
    },
  },
});
```

### Disabled

```typescript
const app = await createApp({
  auth: false,
});
```

## Request Context

Authenticated user available in controllers via `req.user`:

```typescript
class ProfileController extends BaseController {
  async getProfile(req: IRequestContext): Promise<IControllerResponse> {
    if (!req.user) return { success: false, error: 'Unauthorized', status: 401 };
    const profile = await this.repository.findById(req.user._id);
    return { success: true, data: profile };
  }
}
```

## Token Revocation

Arc provides the `isRevoked` primitive — you implement the store (Redis, DB, Better Auth):

```typescript
auth: {
  type: 'jwt',
  jwt: { secret: process.env.JWT_SECRET },
  isRevoked: async (decoded) => {
    return await redis.sismember('revoked-tokens', decoded.jti ?? decoded.id);
  },
}
```

**Fail-closed:** If the revocation check throws, the token is rejected.

## Custom Token Extraction

Override how tokens are resolved from the request:

```typescript
auth: {
  type: 'jwt',
  jwt: { secret },
  tokenExtractor: (req) => req.cookies?.['auth-token'] ?? null,
}
```

Default: reads `Authorization: Bearer <token>` header.

## Request Scope

Every authenticated request gets a `request.scope` — a discriminated union
describing the caller's identity. **Five kinds**, all opt-in:

```typescript
import {
  getUserId, getUserRoles, getOrgId, getServiceScopes,
  getScopeContext, getAncestorOrgIds,
  isMember, isService, isElevated, hasOrgAccess,
} from '@classytic/arc/scope';

// Variants:
// { kind: 'public' }
// { kind: 'authenticated', userId?, userRoles? }
// { kind: 'member',   userId?, userRoles, organizationId, orgRoles, teamId?, context?, ancestorOrgIds? }
// { kind: 'service',  clientId, organizationId, scopes?, context?, ancestorOrgIds? }
// { kind: 'elevated', userId?, organizationId?, elevatedBy, context?, ancestorOrgIds? }

const userId   = getUserId(request.scope);                      // string | undefined
const roles    = getUserRoles(request.scope);                   // string[]
const orgId    = getOrgId(request.scope);                       // string | undefined
const oauthSc  = getServiceScopes(request.scope);               // readonly string[] (service kind)
const branch   = getScopeContext(request.scope, 'branchId');    // string | undefined (custom dimension)
const parents  = getAncestorOrgIds(request.scope);              // readonly string[] (hierarchy)
```

The `service` kind covers API-key / machine-to-machine auth (your custom
auth check installs it via `PermissionResult.scope`). `context` and
`ancestorOrgIds` are populated by your auth function for app-defined
tenancy dimensions and parent-child org hierarchies — see the
[Org Module guide](./org) for the full surface.

## Role Hierarchy

Define inheritance so you don't list every admin tier in 20+ permission definitions:

```typescript
import { createRoleHierarchy } from '@classytic/arc/permissions';

const hierarchy = createRoleHierarchy({
  superadmin: ['admin'],
  admin: ['branch_manager'],
  branch_manager: ['staff'],
});

hierarchy.expand(['admin']);                    // → ['admin', 'branch_manager', 'staff']
hierarchy.includes(['admin'], 'staff');        // → true
```
