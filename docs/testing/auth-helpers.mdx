# Better Auth Test Helpers

Composite test-setup helpers for arc apps using Better Auth. Sits on top of the [`TestAuthProvider`](./auth-session) primitive so you get the session-level pattern (`auth.as('admin').headers`) out of the box while skipping the ~150 LOC of `fastify.inject` orchestration every Better Auth test setup otherwise needs.

Two entry points:

- **`setupBetterAuthTestApp(config)`** — one call for the common case: signup N users, create an org, add non-creator users as members, set active org, return a ready-to-use `TestAuthProvider`.
- **`createBetterAuthTestHelpers(opts?)`** — stateless flow-level helpers for when the orchestrator doesn't fit (multi-org setups, custom invitation flows, staged user creation).

Plus **`safeParseBody`** for handling Better Auth's occasional empty 204 responses without crashes.

```typescript
import {
  setupBetterAuthTestApp,
  createBetterAuthTestHelpers,
  safeParseBody,
} from '@classytic/arc/testing';
```

## `setupBetterAuthTestApp` — composite setup

```typescript
import { beforeAll, afterAll } from 'vitest';
import { createTestApp, setupBetterAuthTestApp, expectArc } from '@classytic/arc/testing';
import { auth as betterAuthInstance } from '../src/auth.js';
import { createBetterAuthAdapter } from '@classytic/arc/auth';
import jobResource from '../src/resources/job/job.resource.js';

let ctx: Awaited<ReturnType<typeof setupBetterAuthTestApp>>;

beforeAll(async () => {
  // 1. Build the app with Better Auth wired up.
  const testApp = await createTestApp({
    resources: [jobResource],
    authMode: 'better-auth',
    auth: {
      type: 'betterAuth',
      betterAuth: createBetterAuthAdapter({ auth: betterAuthInstance }),
    },
  });

  // 2. Hand it to the orchestrator.
  ctx = await setupBetterAuthTestApp({
    app: testApp.app,
    org: { name: 'Test Corp', slug: 'test-corp' },
    users: [
      { key: 'admin',  email: 'admin@test.com',  password: 'pw', name: 'Admin',  role: 'admin',  isCreator: true },
      { key: 'member', email: 'member@test.com', password: 'pw', name: 'Member', role: 'member' },
      { key: 'viewer', email: 'viewer@test.com', password: 'pw', name: 'Viewer', role: 'viewer' },
    ],
    addMember: async ({ app, creatorToken, orgId, userId, role }) => {
      // Better Auth's org plugin offers multiple member-add paths
      // (invitation vs direct add). Your app knows which one it uses.
      return app.inject({
        method: 'POST',
        url: '/api/auth/organization/add-member',
        payload: { organizationId: orgId, userId, role },
        headers: {
          'content-type': 'application/json',
          authorization: `Bearer ${creatorToken}`,
        },
      });
    },
  });
});

afterAll(() => ctx.teardown());

it('admin can list jobs', async () => {
  const res = await ctx.app.inject({
    method: 'GET',
    url: '/jobs',
    headers: ctx.auth.as('admin').headers,
  });
  expectArc(res).ok();
});

it('viewer cannot create jobs', async () => {
  const res = await ctx.app.inject({
    method: 'POST',
    url: '/jobs',
    payload: { title: 'denied' },
    headers: ctx.auth.as('viewer').headers,
  });
  expectArc(res).forbidden();
});
```

### What it does, in order

1. **Signs up every user** via `POST {basePath}/sign-up/email`. Preserves tokens + user ids per key.
2. **Creator creates the org** via `POST {basePath}/organization/create`. Exactly one user MUST be `isCreator: true` — zero or multiple throws at setup time (ambiguous ownership).
3. **Adds non-creator users** via your `addMember` callback. Skipped when `addMember` is omitted (caller wires membership themselves).
4. **Sets active org** on every user via `POST {basePath}/organization/set-active` — Better Auth plugins typically require this before org-scoped endpoints accept the session.
5. **Registers each user** into a fresh `TestAuthProvider`, keyed by `user.key`. The `2.11` pattern works out of the box:
   ```typescript
   const admin = ctx.auth.as('admin');
   await app.inject({ url: '/foo', headers: admin.headers });
   ```

### Returned shape

```typescript
interface SetupBetterAuthTestAppResult {
  app: FastifyInstance;
  orgId: string;                            // org the creator made
  users: Record<string, {                   // keyed by user.key
    userId: string;
    token: string;
    email: string;
    role?: string;
  }>;
  auth: TestAuthProvider;                   // pre-populated with every user
  teardown(): Promise<void>;                // closes the app
}
```

## `createBetterAuthTestHelpers` — flow-level helpers

Use when the orchestrator doesn't fit — multi-org setups, custom invitation flows, staged user creation across tests.

```typescript
const auth = createBetterAuthTestHelpers();
// Or with a custom base path (apps that mount Better Auth under /auth):
const authAlt = createBetterAuthTestHelpers({ basePath: '/auth' });

const signup  = await auth.signUp(app, { email: 'a@x.com', password: 'pw', name: 'Alice' });
const signin  = await auth.signIn(app, { email: 'a@x.com', password: 'pw' });
const org     = await auth.createOrg(app, signup.token, { name: 'Acme' });
await auth.setActiveOrg(app, signup.token, org.orgId);

const headers = auth.authHeaders(signup.token, org.orgId);
// { authorization: 'Bearer ...', 'x-organization-id': 'org-...' }
```

Each helper is stateless — takes the app as a positional argument, so one helper instance works across multiple apps in the same suite.

**Version drift tolerance** — `signUp` / `signIn` look for the token under `.token`, `.session.token`, and `.data.token`; the user id under `.user.id`, `.data.id`, etc. A minor-version Better Auth bump that shuffles the response shape won't break your tests.

## `safeParseBody`

```typescript
const body = safeParseBody<{ token: string }>(res.body);
// → { token: '...' } on valid JSON
// → null on empty / undefined / malformed
```

Better Auth's `set-active` and similar mutations sometimes return an empty 204 body. `JSON.parse('')` throws; `safeParseBody` returns `null`.

## Migration from pre-v2.11 `createBetterAuthTestHelpers` + `setupBetterAuthOrg`

The pre-v2.11 helpers were removed in the 2.11 testing rewrite, then re-shipped in 2.11.x after field reports showed downstream adoption. The new API is a close superset:

| Pre-v2.11 | v2.11.x replacement |
|---|---|
| `createBetterAuthTestHelpers()` | Same name, same surface — unchanged |
| `setupBetterAuthOrg({ createApp, ... })` | `setupBetterAuthTestApp({ app, ... })` — takes a built app instead of a `createApp` callback |
| `safeParseBody` | Same name, same signature — unchanged |

Key difference: `setupBetterAuthTestApp` takes an already-built `app` (you call `createTestApp` separately) rather than a `createApp` callback. This keeps the orchestrator focused on the Better Auth flow rather than app-lifecycle coordination. Composes cleanly with [`createTestApp`](./test-app).

## API reference

### `setupBetterAuthTestApp(input)`

```typescript
interface SetupBetterAuthTestAppInput {
  app: FastifyInstance;
  org: { name: string; slug?: string; metadata?: Record<string, unknown> };
  users: ReadonlyArray<{
    key: string;
    email: string;
    password: string;
    name: string;
    role?: string;
    isCreator?: boolean;
  }>;
  addMember?: (data: {
    app: FastifyInstance;
    creatorToken: string;
    orgId: string;
    userId: string;
    role: string;
  }) => Promise<{ statusCode: number; body: string }>;
  basePath?: string; // default '/api/auth'
}
```

### `createBetterAuthTestHelpers(options?)`

```typescript
interface BetterAuthTestHelpers {
  signUp(app, { email, password, name }): Promise<{ statusCode, token, userId, body }>;
  signIn(app, { email, password }): Promise<{ statusCode, token, userId, body }>;
  createOrg(app, token, { name, slug?, metadata? }): Promise<{ statusCode, orgId, body }>;
  setActiveOrg(app, token, orgId): Promise<{ statusCode, body }>;
  authHeaders(token, orgId?): Record<string, string>;
}
```

### `safeParseBody<T>(body)`

```typescript
function safeParseBody<T = unknown>(body: string | undefined): T | null;
```
