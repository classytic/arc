# Events Module

Domain event system with pluggable transports for external integration.

## Hooks vs Events

Arc has two complementary event systems:

| Aspect | Hooks | Events |
|--------|-------|--------|
| **Purpose** | Internal lifecycle callbacks | External integration layer |
| **Scope** | Same process, synchronous flow | Cross-service, async by design |
| **Use when** | Validating, transforming, auditing within service | Notifying other services, building event-driven systems |
| **Transport** | In-process only | Pluggable (Memory → Redis/RabbitMQ/Kafka) |
| **Pattern** | `beforeCreate`, `afterUpdate` | `product.created`, `order.updated` |

**Rule of thumb:**
- **Hooks** = internal domain logic (validation, cache invalidation, derived data)
- **Events** = public integration (notify external subscribers, event sourcing)

## Auto-Emitted Events

`BaseController` automatically emits events on CRUD operations when the event plugin is registered:

| Operation | Event Type | Payload |
|-----------|------------|---------|
| `create()` | `{resource}.created` | Created document |
| `update()` | `{resource}.updated` | Updated document |
| `delete()` | `{resource}.deleted` | Deleted document |
| `restore()` | `{resource}.restored` | Restored document |

```typescript
// Automatic - no code needed!
// When you create a product via POST /products:
// → Event 'product.created' is published automatically

// Subscribe to auto-emitted events
await fastify.events.subscribe('product.created', async (event) => {
  console.log('New product:', event.payload.name);
  await indexForSearch(event.payload);
});
```

To disable auto-emission for a specific controller:

```typescript
class ProductController extends BaseController {
  constructor() {
    super({ disableEvents: true });
  }
}
```

## Setup

The `createApp()` factory auto-registers `eventPlugin` — no manual registration needed:

```typescript
import { createApp } from '@classytic/arc/factory';

// Development (in-memory, default — zero config)
const app = await createApp({ preset: 'development' });

// Production (Redis transport)
import { RedisEventTransport } from '@classytic/arc/events/redis';

const app = await createApp({
  preset: 'production',
  stores: {
    events: new RedisEventTransport(redis),  // transport passed to eventPlugin
  },
  arcPlugins: {
    events: {                  // eventPlugin options (omit `transport` — sourced from stores.events)
      logEvents: true,
      retry: { maxRetries: 3, backoffMs: 1000 },
    },
  },
});
```

**Manual registration** (for apps not using `createApp`):

```typescript
import { eventPlugin } from '@classytic/arc/events';

await fastify.register(eventPlugin);                          // Memory transport
await fastify.register(eventPlugin, { transport: redisTransport });  // Custom transport
```

**Disable events entirely:**

```typescript
const app = await createApp({
  arcPlugins: { events: false },  // No eventPlugin, no fastify.events
});
```

## Configuration

```typescript
interface EventPluginOptions {
  transport?: EventTransport;  // Default: MemoryEventTransport
  logEvents?: boolean;         // Log event publishing (default: false)
  failOpen?: boolean;          // Suppress transport errors (default: true)
  retry?: {                    // Auto-retry failed handlers
    maxRetries?: number;       // Default: 3
    backoffMs?: number;        // Initial delay (default: 1000)
    maxBackoffMs?: number;     // Max delay (default: 30000)
    jitter?: number;           // Randomization factor (default: 0.1)
  };
  deadLetterQueue?: {          // For events that exhaust retries
    store?: (event, errors) => Promise<void>; // Custom store (default: publishes to '$deadLetter')
  };
  wal?: {                      // Write-Ahead Log for at-least-once delivery
    save: (event) => Promise<void>;        // Persist before publish
    acknowledge?: (eventId) => Promise<void>; // Mark as delivered
  };
  registry?: EventRegistry;    // Event registry for schema validation + introspection
  validateMode?: 'warn' | 'reject' | 'off'; // How to handle validation failures (default: 'warn' when registry provided)
  onPublish?: (event) => void;    // Callback after successful publish (metrics)
  onPublishError?: (event, error) => void; // Callback on publish failure (alerting)
}
```

**`failOpen` behavior:**
- `true` (default): transport errors are logged and suppressed — publish/subscribe never throw.
  Safe for analytics and non-critical side effects. **Warning:** a transport outage silently
  drops events while requests continue succeeding. Pair with `onPublishError` for monitoring.
- `false`: transport errors propagate to caller — use for business-critical events (billing,
  notifications) where silent loss is unacceptable.

> **Best practice:** Set `failOpen: false` for business events and use `wal` for at-least-once
> delivery. Use `failOpen: true` (default) only for fire-and-forget analytics.

## Retry & Dead Letter Queue

Enable automatic retry with exponential backoff for failed event handlers:

```typescript
await fastify.register(eventPlugin, {
  transport: new RedisEventTransport({ client: redis }),
  retry: {
    maxRetries: 3,
    backoffMs: 1000,      // 1s → 2s → 4s (exponential)
    maxBackoffMs: 30000,
  },
  deadLetterQueue: {},    // Failed events → '$deadLetter' topic
});
```

When a handler throws, the retry wrapper re-invokes it with exponential backoff. After exhausting all retries, the event is published to the `$deadLetter` event type (or your custom store):

```typescript
// Monitor dead letters
await fastify.events.subscribe('$deadLetter', async (event) => {
  logger.error({ event }, 'Event handler exhausted all retries');
  await alertOps(event);
});
```

**Custom dead letter store:**

```typescript
await fastify.register(eventPlugin, {
  retry: { maxRetries: 3, backoffMs: 1000 },
  deadLetterQueue: {
    store: async (event, errors) => {
      await db.collection('dead_letters').insertOne({ event, errors, createdAt: new Date() });
    },
  },
});
```

**When NOT to use retry:** For fire-and-forget handlers (e.g., analytics), catch errors in the handler itself. Retry is best for operations that must eventually succeed (emails, webhooks).

## defineEvent — Typed Events with Schema Validation

Declare events with schemas for runtime validation and introspection:

```typescript
import { defineEvent, createEventRegistry } from '@classytic/arc/events';

const OrderCreated = defineEvent({
  name: 'order.created',
  version: 1,
  description: 'Emitted when an order is placed',
  schema: {
    type: 'object',
    properties: {
      orderId: { type: 'string' },
      total: { type: 'number' },
      currency: { type: 'string' },
    },
    required: ['orderId', 'total'],
  },
});

// Type-safe event creation with auto-generated metadata
const event = OrderCreated.create(
  { orderId: 'o-1', total: 100 },
  { userId: 'user-1' },
);
await fastify.events.publish(event.type, event.payload, event.meta);
```

### Event Registry

Catalog events and auto-validate payloads on publish:

```typescript
const registry = createEventRegistry();
registry.register(OrderCreated);
registry.register(OrderShipped);

// Wire into eventPlugin
const app = await createApp({
  arcPlugins: {
    events: { registry, validateMode: 'warn' },
  },
});
```

**Validate modes:**

| Mode | Behavior |
|------|----------|
| `'warn'` | Log warning on invalid payload, still publish (default when registry provided) |
| `'reject'` | Throw error on invalid payload, do NOT publish |
| `'off'` | Skip validation — registry is introspection-only |

Unregistered events always pass validation — the registry is fully opt-in.

**Runtime introspection:**

```typescript
// Get event catalog at runtime
const catalog = app.events.registry?.catalog();
// → [{ name: 'order.created', version: 1, schema: {...}, description: '...' }]

// Validate a payload manually
const result = registry.validate('order.created', payload);
if (!result.valid) console.error(result.errors);
```

**CLI introspection:** Export the registry from your entry file and `arc describe` auto-detects it:

```typescript
// src/events.ts
export const eventRegistry = createEventRegistry();
eventRegistry.register(OrderCreated);
```

```bash
arc describe ./dist/index.js --pretty
# Output includes: eventCatalog: [{ name, version, hasSchema, schemaFields, requiredFields }]
```

### Schema Evolution

Use `version` to evolve event schemas without breaking existing consumers:

```typescript
const OrderCreatedV1 = defineEvent({
  name: 'order.created',
  version: 1,
  schema: { type: 'object', properties: { orderId: { type: 'string' } }, required: ['orderId'] },
});

const OrderCreatedV2 = defineEvent({
  name: 'order.created',
  version: 2,
  schema: { type: 'object', properties: { orderId: { type: 'string' }, currency: { type: 'string' } }, required: ['orderId', 'currency'] },
});

registry.register(OrderCreatedV1);
registry.register(OrderCreatedV2); // Both coexist — validation uses latest version
```

## Transports

Arc ships three built-in transports:

| Transport | Import | Ordering | Durability |
|-----------|--------|----------|------------|
| `MemoryEventTransport` | `@classytic/arc/events` | Sequential (awaited) | None (lost on restart) |
| `RedisEventTransport` | `@classytic/arc/events/redis` | Unordered (fire-and-forget) | None (pub/sub) |
| `RedisStreamTransport` | `@classytic/arc/events/redis-stream` | Ordered (consumer groups) | Durable (at-least-once) |

### Memory Transport (Default)

```typescript
import { MemoryEventTransport } from '@classytic/arc/events';

await fastify.register(eventPlugin, {
  transport: new MemoryEventTransport(),
});
```

- Handlers execute **sequentially** with `await`
- One failing handler does NOT block others
- **Use for:** Development, testing, single-instance

### Redis Pub/Sub Transport

```typescript
import { RedisEventTransport } from '@classytic/arc/events/redis';
import Redis from 'ioredis';

const redis = new Redis(process.env.REDIS_URL);
const transport = new RedisEventTransport(redis, { channel: 'arc-events' });
```

- Fan-out delivery to ALL connected subscribers
- Handlers fire concurrently (unordered)
- Events missed during subscriber downtime are lost
- **Use for:** Real-time notifications, cache invalidation, multi-instance

### Redis Streams Transport

```typescript
import { RedisStreamTransport } from '@classytic/arc/events/redis-stream';

const transport = new RedisStreamTransport(redis, {
  stream: 'arc-events',
  consumerGroup: 'my-service',
});
```

- Ordered delivery via consumer groups
- Events persisted and acknowledged (**at-least-once** — not exactly-once)
- Failed messages are auto-reclaimed after `claimTimeoutMs`, which can cause duplicate delivery
- Consumers **must be idempotent** — use `event.meta.id` as a deduplication key
- Load balancing across instances in same consumer group
- **Use for:** Ordered processing, durable event logs

### Custom Transport

Implement `EventTransport` interface for any message broker (RabbitMQ, Kafka, NATS, SQS):

```typescript
import type { EventTransport, DomainEvent, EventHandler } from '@classytic/arc/events';

class NatsEventTransport implements EventTransport {
  readonly name = 'nats';

  async publish(event: DomainEvent): Promise<void> {
    await this.nats.publish(event.type, JSON.stringify(event));
  }

  async subscribe(pattern: string, handler: EventHandler): Promise<() => void> {
    const sub = this.nats.subscribe(pattern);
    (async () => { for await (const msg of sub) { await handler(JSON.parse(msg.data)); } })();
    return () => sub.unsubscribe();
  }

  async close(): Promise<void> { await this.nats.close(); }
}
```

## Multi-Instance Deployment

Pick a transport by what you need. Anything not in [Transports](#transports) above is a custom transport — arc's `EventTransport` interface has 3 methods (`publish`, `subscribe`, `close`); drop in Kafka / RabbitMQ / SQS / NATS with ~100 LOC per broker. Same [`Custom Transport` pattern](#custom-transport), same wiring — `stores: { events: new YourTransport(...) }`.

| Broker | Use case |
|---|---|
| RabbitMQ | Guaranteed delivery, queuing |
| Kafka | High-throughput streaming |
| AWS SQS/SNS | Serverless, managed |
| NATS | Lightweight pub/sub |

For ordered + durable processing across instances, the built-in `RedisStreamTransport` is the no-new-infra choice; bring Kafka only when you've outgrown it.

## Event Outbox Pattern

At-least-once delivery guarantee. Store events in the same DB transaction as your write, relay to transport asynchronously.

Two ways to wire the store:

```typescript
// 1. Repository (recommended — mongokit / prismakit / your own kit)
import { EventOutbox } from '@classytic/arc/events';
import {
  Repository,
  methodRegistryPlugin,
  batchOperationsPlugin,
} from '@classytic/mongokit';
import mongoose, { Schema } from 'mongoose';

const OutboxModel = mongoose.model(
  'ArcOutbox',
  new Schema({}, { strict: false, timestamps: false, _id: false }),
  'event_outbox',
);

const outbox = new EventOutbox({
  repository: new Repository(OutboxModel, [
    methodRegistryPlugin(),
    batchOperationsPlugin(), // required — outbox.purge() calls deleteMany
  ]),
  transport: redisTransport,
});

// 2. Direct store (dev, or a custom implementation)
import { EventOutbox, MemoryOutboxStore } from '@classytic/arc/events';

const outbox = new EventOutbox({
  store: new MemoryOutboxStore(),
  transport: redisTransport,
});

// In your business logic (same transaction as DB write)
await outbox.store({
  type: 'order.created',
  payload: { orderId: '123' },
  meta: { id: crypto.randomUUID(), timestamp: new Date() },
});

// Relay cron (runs every few seconds)
const relayed = await outbox.relay();  // publishes pending → transport
```

When you pass `repository`, arc adapts it to the `OutboxStore` contract
internally — `create` / `findAll` / `deleteMany` / `findOneAndUpdate`
cover save, claim, ack, fail, and DLQ reads. Memory store is still
available for dev and as a full reference implementation of the contract.

### Required plugins for mongokit-backed outbox

| Method used by adapter | Source |
|------------------------|--------|
| `create` / `getOne` / `findAll` / `findOneAndUpdate` | base `Repository` |
| `deleteMany` | `batchOperationsPlugin` (+ `methodRegistryPlugin`) |

Without `batchOperationsPlugin`, `EventOutbox` throws at construction
with `"repository is missing required methods: deleteMany"`.

### Why `strict: false`?

Arc owns the stored outbox doc shape (see `repository-outbox-adapter.ts`).
A strict Mongoose schema would drop fields arc adds in future releases.
`strict: false` forwards every field as-is, so arc's contract defines
the on-disk layout.

### Recommended indexes

Declare these at boot — arc creates none automatically. The claim path
is the hot query; indexing it is essential under any real load.

```typescript
// Hot path: claimPending filter — { status, visibleAt, leaseOwner/leaseExpiresAt }
await OutboxModel.collection.createIndex({ status: 1, visibleAt: 1 });

// Purge scan — delivered docs older than cutoff
await OutboxModel.collection.createIndex({ status: 1, deliveredAt: 1 });

// Optional: unique dedupe (when you pass options.dedupeKey to store())
await OutboxModel.collection.createIndex(
  { dedupeKey: 1 },
  { unique: true, partialFilterExpression: { dedupeKey: { $type: 'string' } } },
);
```

**OutboxStore interface** — Arc is DB-agnostic; you implement this for your database.
The contract has three required methods and several optional capabilities that stores
can opt into for richer semantics:

```typescript
interface OutboxStore {
  // ── Required ──────────────────────────────────────────────────────────
  save(event: DomainEvent, options?: OutboxWriteOptions): Promise<void>;
  getPending(limit: number): Promise<DomainEvent[]>;
  acknowledge(eventId: string, options?: OutboxAcknowledgeOptions): Promise<void>;

  // ── Optional capabilities ─────────────────────────────────────────────
  /** Lease-based atomic claim — required for safe multi-worker relay */
  claimPending?(options?: OutboxClaimOptions): Promise<DomainEvent[]>;
  /** Failure tracking — enables retries, backoff, dead-letter */
  fail?(eventId: string, error: OutboxErrorInfo, options?: OutboxFailOptions): Promise<void>;
  /** Purge old acknowledged events — DB-specific cleanup */
  purge?(olderThanMs: number): Promise<number>;
}
```

Arc ships only `MemoryOutboxStore` (for dev/testing) — it's a reference implementation
of the **full** capability set. Production stores are implemented by you — this keeps
the framework DB-agnostic and avoids coupling to any ORM or driver.

### Transactional writes

Pass your DB session through `OutboxWriteOptions.session` to persist the event in
the same transaction as your business write. Stores that support it will honor it;
others ignore the field (best-effort contract).

```typescript
await db.withTransaction(async (session) => {
  await orders.insertOne(order, { session });
  await outbox.store(
    { type: 'order.created', payload: order, meta: { id, timestamp: new Date() } },
    { session },
  );
  // DB commit and event commit are atomic
});
```

Additional write options — all optional, all backward-compatible:

| Option | Purpose |
|--------|---------|
| `session` | DB transaction handle for atomic store-and-write |
| `visibleAt` | Earliest time the event should be visible to relay (delayed publishing) |
| `dedupeKey` | Idempotency key — stores that support it skip duplicates |
| `partitionKey` | Routing key for sharded transports (Kafka, Redis Streams) |
| `headers` | Propagated to the transport layer |

### Multi-worker relay (claim + lease)

If you run multiple relay workers, implement `claimPending` on your store. Each
worker atomically claims a batch with a lease TTL — stale leases are reclaimed
automatically, and no two workers publish the same event.

```typescript
const outbox = new EventOutbox({
  store,
  transport,
  consumerId: `relay-${process.pid}`,
  leaseMs: 30_000, // 30s lease
});

// On every tick, relay() claims its own batch — safe to run on N workers
setInterval(() => outbox.relay(), 1_000);
```

`EventOutbox` automatically detects `claimPending` and uses it over `getPending`.
Stores without it stay single-worker safe.

### Retry and dead-letter (fail hook)

If your store implements `fail`, relay calls it on publish failure instead of
stopping the batch. This unlocks retry scheduling (via `retryAt`) and dead-letter
handling (via `deadLetter: true`):

```typescript
// Inside your store.fail implementation
async fail(eventId, error, options) {
  const entry = await this.findById(eventId);
  entry.attempts++;
  entry.lastError = error;

  if (entry.attempts >= MAX_ATTEMPTS) {
    entry.status = 'dead_letter'; // won't be retried
    return;
  }

  // Exponential backoff
  const backoffMs = Math.min(60_000, 1_000 * 2 ** entry.attempts);
  entry.visibleAt = new Date(Date.now() + backoffMs);
  await this.update(entry);
}
```

Without `fail`, relay keeps its legacy behavior: stop on first failure, leave
the event pending, retry on the next `relay()` call.

### Recommended Mongo store schema

For apps building a Mongo-backed store, this shape captures the full contract:

```typescript
{
  eventId: string,        // unique — from event.meta.id
  type: string,
  payload: unknown,
  meta: unknown,
  status: 'pending' | 'delivered' | 'dead_letter',
  attempts: number,
  leaseOwner: string | null,
  leaseExpiresAt: Date | null,
  visibleAt: Date,        // default: now
  dedupeKey?: string,     // unique sparse index for idempotency
  lastError?: { message: string, code?: string },
  createdAt: Date,
  deliveredAt: Date | null,
}
```

Recommended indexes:
- `{ status: 1, visibleAt: 1 }` — claim scan
- `{ eventId: 1 }` unique — ack/fail lookup
- `{ dedupeKey: 1 }` unique sparse — idempotency
- `{ deliveredAt: 1 }` with `expireAfterSeconds` — automatic purge

### Terminology — `delivered` is canonical

Arc's outbox uses **`delivered`** as the canonical term for the terminal
successful state:

| Term | Meaning |
|------|---------|
| `acknowledge()` | The store operation that transitions an event to delivered |
| `delivered` | The resulting state |
| `deliveredAt` | The timestamp field / column store authors should use |

Older drafts of this doc used `acknowledgedAt` — that is deprecated. Use
**`deliveredAt`** in all new store implementations.

### Outbox Cleanup

Delivered events accumulate over time. Arc provides `purge()` as an **optional contract**
on `OutboxStore` — implement it using your DB's native cleanup mechanism:

| Database | Recommended Cleanup | Notes |
|----------|-------------------|-------|
| MongoDB | TTL index on `deliveredAt` | Automatic, zero-code once index is created |
| PostgreSQL/MySQL | Scheduled `DELETE FROM outbox WHERE status = 'delivered' AND delivered_at < NOW() - INTERVAL '7 days'` | Use pg_cron or app-level scheduler |
| Redis | Key expiry (`EXPIRE`) on delivered entries | Built-in, no scheduler needed |

Only events in the `delivered` state are eligible for purge — `pending`,
`failed`, and `dead_letter` rows must never be removed by cleanup.

```typescript
// Option A: Use Arc's purge() contract (if your store implements it)
const purged = await outbox.purge(7 * 24 * 60 * 60 * 1000); // 7 days
console.log(`Purged ${purged} old outbox events`);

// Option B: Use your DB's native cleanup (no Arc involvement)
// MongoDB: db.outbox.createIndex({ deliveredAt: 1 }, { expireAfterSeconds: 604800 })
// PostgreSQL: CREATE EXTENSION pg_cron; SELECT cron.schedule('0 3 * * *', $$DELETE FROM outbox WHERE status='delivered' AND delivered_at < NOW() - INTERVAL '7 days'$$);
```

> **Arc's boundary:** Arc defines the interface and calls `store.purge()` if implemented.
> It does **not** ship DB-specific outbox stores — that's your app's responsibility, using
> whichever database and cleanup strategy fits your stack.

### Backend guarantees — what you get from different stores

Arc's outbox is intentionally backend-neutral: it specifies a contract, not a
storage technology. The guarantees you actually get depend on the store you
plug in, so choose deliberately:

| Store backend | Transactional with business DB write | Durable across restarts | Atomic multi-worker claim | Notes |
|---|---|---|---|---|
| **Mongo / SQL in same DB as business** | ✅ (when store.save accepts `{ session }` and the host passes the transaction handle) | ✅ | ✅ (via `FOR UPDATE SKIP LOCKED` or `findOneAndUpdate`) | Classic transactional outbox — strongest guarantees |
| **Separate Mongo / SQL database** | ❌ (two-phase commit needed — don't) | ✅ | ✅ | Use only if you accept eventual consistency between business and events |
| **Redis Streams / Redis hash** | ❌ | ✅ (with AOF/RDB) | ✅ (via consumer groups) | Durable but NOT atomic with your business DB — handle the race |
| **Kafka / RabbitMQ direct producer** | ❌ | ✅ | ✅ | Treat as transport, not store — skip the outbox layer if you're OK with fire-and-forget |
| **SQS / SNS** | ❌ | ✅ | ✅ (SQS visibility timeout) | Handle the host-coordinates case: buffer in DB, then forward to SQS |

**Key rule**: the only way to get true transactional outbox semantics is for
the store to share the same DB transaction as your business write. Pass the
transaction handle through `OutboxWriteOptions.session` and verify the store
honors it. Any other topology gives you durability without atomicity — and
that's a conscious trade-off, not a bug.

### Retry helper — `exponentialBackoff()`

Store authors implementing `fail()` can use Arc's `exponentialBackoff()` to
compute consistent `retryAt` timestamps with jitter:

```typescript
import { exponentialBackoff } from '@classytic/arc/events';

async fail(eventId, error, options) {
  const entry = await this.findById(eventId);
  entry.attempts++;

  if (entry.attempts >= MAX_ATTEMPTS) {
    entry.status = 'dead_letter';
    await this.update(entry);
    return;
  }

  // 1s, 2s, 4s, 8s, 16s, 32s, 60s (capped), with ±20% jitter
  entry.visibleAt = exponentialBackoff({ attempt: entry.attempts });
  entry.status = 'pending';
  await this.update(entry);
}
```

Options: `baseMs` (first delay, default 1000), `maxMs` (cap, default 60000),
`jitter` (0–1, default 0.2). Jitter is always additive — never schedules
earlier than the computed delay — so it safely spreads retry bursts across
multiple workers without risking earlier-than-intended publishes.

### Batched publish — `publishMany`

Transports that can efficiently batch (Kafka producer, Redis pipeline,
RabbitMQ publisher confirms, SQS `SendMessageBatch`) should implement the
optional `publishMany(events)` method. When present, `EventOutbox.relay`
auto-detects it and sends the entire batch in one network call instead of
N per-event calls. Disable per-instance via `usePublishMany: false`.

```typescript
class KafkaEventTransport implements EventTransport {
  async publishMany(events) {
    const results = new Map<string, Error | null>();
    const kafkaBatch = events.map((e) => ({
      topic: e.type,
      messages: [{ key: e.meta.id, value: JSON.stringify(e) }],
    }));
    const outcomes = await producer.sendBatch({ topicMessages: kafkaBatch });
    // Map per-message outcomes back to event IDs
    for (const ev of events) {
      results.set(ev.meta.id, null); // or Error if that message failed
    }
    return results;
  }
}
```

### Rich relay results — `relayBatch()`

For operational dashboards and alerting, `relayBatch()` returns a
`RelayResult` with per-kind counts instead of just the success total:

```typescript
const result = await outbox.relayBatch();
//   {
//     relayed: 87,
//     attempted: 100,
//     publishFailed: 10,
//     ackFailed: 1,
//     ownershipMismatches: 2,
//     malformed: 0,
//     failHookErrors: 0,
//     usedPublishMany: true,
//   }

metrics.increment('outbox.relayed', result.relayed);
metrics.increment('outbox.publish_failed', result.publishFailed);
metrics.increment('outbox.ownership_mismatch', result.ownershipMismatches);

if (result.malformed > 0) {
  logger.error('Store returned malformed event — investigate corrupted rows');
}
```

`relay()` (returning just the count) remains the simple shortcut for workers
that don't need the breakdown.

## Webhooks (Outbound + Inbound Verification)

Auto-dispatch Arc events to customer webhook endpoints with HMAC-SHA256 signing and bounded concurrency.

```typescript
import { webhookPlugin, verifySignature } from '@classytic/arc/integrations/webhooks';

// Outbound: auto-dispatch events to subscribers
await fastify.register(webhookPlugin, { concurrency: 5 });

await app.webhooks.register({
  id: 'wh-1',
  url: 'https://customer.com/webhook',
  events: ['order.created', 'order.*'],
  secret: 'whsec_abc123',
});

await app.events.publish('order.created', { orderId: '123' });
// → POST https://customer.com/webhook with x-webhook-signature header

// Inbound: verify signatures from other Arc services or third parties
fastify.post('/webhooks/incoming', async (req, reply) => {
  if (!verifySignature(req.rawBody, secret, req.headers['x-webhook-signature'])) {
    return reply.status(401).send({ error: 'Invalid signature' });
  }
  // handle event
});
```

See [integrations reference](/docs/integrations) for WebhookStore, delivery log, custom algorithms, and full verification options.
