# Factory — createApp()

Secure, production-ready Fastify app creation with sensible defaults and **opt-out security**.

## Quick Start

```typescript
import { createApp, loadResources } from '@classytic/arc/factory';
import mongoose from 'mongoose';

await mongoose.connect(process.env.MONGO_URI);

const app = await createApp({
  preset: 'production',
  resourcePrefix: '/api/v1',
  resources: await loadResources(import.meta.url),
  auth: { type: 'jwt', jwt: { secret: process.env.JWT_SECRET } },
  cors: { origin: [process.env.FRONTEND_URL] },
});

await app.listen({ port: 3000, host: '0.0.0.0' });
```

### Resource Registration

Four ways to register resources (pick one):

```typescript
// Auto-discover from directory (recommended for large apps)
const app = await createApp({
  resources: await loadResources('./src/resources'),
});

// Explicit array (clear for small apps)
const app = await createApp({
  resources: [productResource, orderResource],
});

// Factory function — runs AFTER bootstrap, before route wiring
// (canonical answer to "my repository lives in an async-booted engine")
const app = await createApp({
  bootstrap: [async () => { await ensureCatalogEngine(); }],
  resources: async () => {
    const cat = await ensureCatalogEngine();
    return [
      defineResource({
        name: 'product',
        adapter: createMongooseAdapter(cat.models.Product, cat.repositories.product),
        // ...
      }),
    ];
  },
});

// Via plugins callback (full Fastify control)
const app = await createApp({
  plugins: async (fastify) => {
    await fastify.register(productResource.toPlugin());
  },
});
```

### Engine-bound resources via `loadResources({ context })` (2.11.1)

When you have engine-owned resources (catalog, flow, loyalty, …), the
combination of `loadResources({ context })` + factory-style default exports
lets auto-discovery thread the engine into resources without parallel
factory files or `exclude: [...]` bookkeeping:

```typescript
// resources/catalog/category.resource.ts — factory default export
import type { AppContext } from '#core/app/context.js';
export default (ctx: AppContext) =>
  defineResource({
    name: 'category',
    adapter: createMongooseAdapter(
      ctx.catalog.models.Category,
      ctx.catalog.repositories.category,
    ),
    permissions: { /* ... */ },
  });

// app.ts — single sweep, no excludes, no parallel factories
const app = await createApp({
  resources: async () => {
    const [catalog, flow] = await Promise.all([
      ensureCatalogEngine(),
      ensureFlowEngine(),
    ]);
    return loadResources<AppContext>(import.meta.url, {
      context: { catalog, flow },
    });
  },
});
```

**Detection**: `loadResources` checks `typeof default === 'function'`.
`defineResource()` returns a class instance (`typeof === 'object'`), so a
function-typed default unambiguously means "factory — call me with
`options.context`." Plain `ResourceLike` exports keep working unchanged.
Async factories are awaited.

A factory that returns a non-resource value (or throws) is reported via
the injected `logger.warn` channel — distinct from the regular "skipped
file" path so misconfigurations surface clearly.

### Strict resource flags

Two independent flags catch separate failure modes:

| Flag | What it catches | Production preset default |
|------|-----------------|---------------------------|
| `strictResourceDir` | `resourceDir` resolved to zero matching files (typo'd path / stale `dist/`) | **`true`** |
| `strictResources` | Two resources share the same `name` (stale `dist/` collisions, duplicate registration) | **`true`** |

**You don't need to set them manually when using `preset: 'production'`** —
both default to `true` automatically. Setting `strictResourceDir: config.isProduction` /
`strictResources: config.isProduction` in your factory is redundant once the production preset is on.

Override either explicitly only when you're opting *out* — e.g. for a hot-path migration where you genuinely expect duplicate names temporarily.

### Logging — arcLog fallback + injection override

`loadResources` follows arc's standard logger convention: warnings flow
through `arcLog('loadResources')` by default (same channel as every other
arc-internal warn), and an injected `logger` overrides the fallback for
that call. Three control points:

```typescript
// 1. Global suppression (silences ALL arc warnings, not just loadResources)
process.env.ARC_SUPPRESS_WARNINGS = '1';

// 2. Per-call override — route to your own logger
import pino from 'pino';
const logger = pino();
const resources = await loadResources(import.meta.url, {
  context: { catalog },
  logger: { warn: (msg) => logger.warn(msg) },
});

// 3. Per-call no-op — discard warnings without setting env
await loadResources(import.meta.url, { logger: { warn: () => undefined } });

// 4. Custom arcLog sink — routes EVERY arc warning (loadResources + others)
import { configureArcLogger } from '@classytic/arc';
configureArcLogger({ writer: pinoAdapter });
```

> **Migration from pre-2.11.1**: the `silent: boolean` option was removed.
> The default also changed: pre-2.11.1 was silent-when-no-logger; 2.11.1
> routes through arcLog (warn-by-default). Migration:
>
> - `silent: true` → `ARC_SUPPRESS_WARNINGS=1` env var, or pass `logger: { warn: () => undefined }` per-call.
> - `silent: !isDev` → set `ARC_SUPPRESS_WARNINGS=1` in production env.
> - Callers that already injected a `logger` → no change (injection still wins over the fallback).
> - Callers that relied on the silent default → warnings will now show. This is intentional — skip / factory-failure events used to disappear silently, which masked misconfiguration. Use one of the three suppression knobs above if you genuinely want silence.

`loadResources` options: `exclude`, `include`, `suffix` (default: `.resource`), `recursive` (default: true), `context`, `logger`.

**When to use the factory form** — resource definitions are evaluated at module-import time, so any resource whose adapter depends on an async-booted engine (e.g. `@classytic/catalog`, `@classytic/flow` patterns using `await createX(...)`) can't be defined statically. Before 2.11.x, hosts wrote per-resource lazy-bridge adapters that awaited `ensureEngine()` on every CRUD call — pure boilerplate. The factory slot runs between `bootstrap[]` and route wiring, so `defineResource(...)` calls inside it see the engine already live.

**Contract** — an explicit `resources` option always wins over `resourceDir` auto-discovery, including when it's an empty array OR a factory returning `[]`. Use `resources: []` or `resources: () => []` to intentionally disable resource loading (test subprocesses, CLI jobs, health-check stubs).

### Boot Order

```
1. Arc core (security, auth, events)
2. plugins()                  ← infra (DB, SSE, docs)
3. bootstrap[]                ← domain init (engines, singletons)
4. resources factory (if any) ← resolved here — engine state is live
5. resources[]                ← routes (under resourcePrefix, split by skipGlobalPrefix)
6. afterResources()           ← post-registration wiring
7. onReady()                  ← Fastify ready
```

```typescript
const app = await createApp({
  preset: 'production',
  resourcePrefix: '/api/v1',

  plugins: async (f) => {
    await mongoose.connect(process.env.MONGO_URI);
    await f.register(docsPlugin);
  },
  bootstrap: [inventoryInit, accountingInit, loyaltyInit],
  resources: await loadResources(import.meta.url),
  afterResources: async (f) => { subscribeEvents(f); },

  auth: { type: 'jwt', jwt: { secret: process.env.JWT_SECRET } },
});
```

### Resource Prefix & Per-Resource Override

```typescript
// All resources under /api/v1
const app = await createApp({
  resourcePrefix: '/api/v1',
  resources: [productResource, orderResource, webhookResource],
});
// product → /api/v1/products, order → /api/v1/orders

// Webhook opts out of the global prefix
defineResource({
  name: 'webhook',
  prefix: '/hooks',
  skipGlobalPrefix: true,   // → /hooks (not /api/v1/hooks)
});
```

### Import Compatibility

`loadResources()` uses dynamic `import()` at runtime. This works with **standard relative imports** and **Node.js `#` subpath imports** (defined in `package.json` `imports`), but **not** with TypeScript-only path aliases.

| Import style in resource files | Works? | Why |
|------|--------|-----|
| `./helper.js`, `../shared/utils.js` | **Yes** | Real relative paths — `.js→.ts` resolved by vitest/tsx |
| `#shared/utils.js`, `#lib/db.ts` | **Yes** | Node.js native — resolved via `package.json` `imports` field |
| `@/utils/db`, `~/lib/auth` | **No** | tsconfig `paths` — compile-time only, Node.js ignores them |

**If your project uses tsconfig path aliases** (`@/*`, `~/`), use explicit imports instead:

```typescript
import productResource from './resources/product/product.resource.js';
import orderResource from './resources/order/order.resource.js';

const app = await createApp({
  resources: [productResource, orderResource],
});
```

**Node.js `#` subpath imports** work because they're part of the ESM spec — both `.ts` and `.js` extensions resolve correctly under vitest/tsx. Configure `package.json` `imports` for your environment:

```jsonc
// package.json — development (point to source)
{
  "imports": {
    "#shared/*": "./src/shared/*",
    "#lib/*": "./src/lib/*"
  }
}
```

```jsonc
// package.json — production (point to compiled output)
{
  "imports": {
    "#shared/*": "./dist/shared/*",
    "#lib/*": "./dist/lib/*"
  }
}
```

> **Tip:** Use [Node.js conditional exports](https://nodejs.org/api/packages.html#conditional-exports) to support both environments in a single `package.json`.

### Vitest + `loadResources()`: when to use `import.meta.glob`

`loadResources()` works in production and in most vitest setups. The two cases where it can fail:

1. **Resources with top-level engine dependencies** — e.g., `getAccountModel()` called at module top-level, where the engine must be initialized via `bootstrap[]` before the module can evaluate. Dynamic import doesn't compose with that ordering.

2. **Deeply nested `.js→.ts` resolution through `node_modules`** — vitest's loader hooks intercept top-level dynamic imports, but resolution chains inside transitive packages may fall through to Node's native resolver.

For these cases, use Vite's compile-time `import.meta.glob` to preload resources statically. Arc provides a helper:

```typescript
// tests/setup/preload-resources.ts
import { preloadResources } from '@classytic/arc/testing';

// Vite rewrites import.meta.glob at transform time into a static map.
// Every match goes through vitest's transform pipeline like any other import.
export const preloadedResources = preloadResources(
  import.meta.glob('../../src/resources/**/*.resource.ts', {
    eager: true,
    import: 'default',  // omit if you use `export const fooResource`
  }),
);
```

```typescript
// src/app.ts — accept an override for tests
import { createApp, loadResources, type ResourceLike } from '@classytic/arc/factory';

export async function buildApp(opts: { resources?: ResourceLike[] } = {}) {
  const resources = opts.resources ?? (await loadResources(import.meta.url));
  return createApp({ resources, /* ... */ });
}
```

```typescript
// tests/integration/foo.test.ts
import { buildApp } from '../../src/app.js';
import { preloadedResources } from '../setup/preload-resources.js';

const app = await buildApp({ resources: preloadedResources });
```

#### Deferred loading (when resources need bootstrap to run first)

If your resources call engine init at module top-level (e.g., `getAccountModel()` from an accounting engine), use the async variant:

```typescript
import { preloadResourcesAsync } from '@classytic/arc/testing';
import { initAccountingEngine } from '../../src/engines/accounting.js';

beforeAll(async () => {
  await initAccountingEngine(); // bootstrap FIRST
});

const resources = await preloadResourcesAsync(
  import.meta.glob('../../src/resources/**/*.resource.ts', {
    import: 'default',
    // no eager: true — modules load AFTER beforeAll
  }),
);
```

> **Why this works:** `import.meta.glob` is rewritten by Vite at transform time into a static map of importers, so every match goes through vitest's loader chain — including nested `.js→.ts` resolution. Arc's `preloadResources()` helper just normalizes the result into the `ResourceLike[]` shape that `createApp({ resources })` expects.

## Auth Configuration

Auth uses a **discriminated union** with `type` field:

```typescript
const app = await createApp({
  // JWT
  auth: { type: 'jwt', jwt: { secret: process.env.JWT_SECRET, expiresIn: '15m' } },

  // Better Auth (SaaS with orgs)
  auth: { type: 'betterAuth', betterAuth: createBetterAuthAdapter({ auth, orgContext: true }) },

  // Custom plugin (must decorate fastify.authenticate)
  auth: { type: 'custom', plugin: myAuthPlugin },

  // Custom function
  auth: { type: 'authenticator', authenticate: async (req, reply) => { ... } },

  // Disabled
  auth: false,
});
```

## Runtime Modes

```typescript
const app = await createApp({
  runtime: 'memory',       // Default — zero config, single instance
  // runtime: 'distributed',  // Multi-instance — requires durable/shared stores

  stores: {                // Events required; cache/queryCache required when enabled
    events: new RedisEventTransport({ client: redis }),
    cache: new RedisCacheStore({ client: redis }),
    queryCache: new RedisCacheStore({ client: redis, prefix: 'arc:qc:' }),
    // idempotency: new RedisIdempotencyStore({ client: redis }), // recommended for write dedupe
  },
});
```

| Mode | Use Case | Requirements |
|------|----------|--------------|
| `memory` | Development, single-instance | None |
| `distributed` | Multi-instance production | Durable events transport, plus shared stores for enabled plugins/features |

Fail-fast validation: using `distributed` mode without required shared stores throws immediately for enforced subsystems.
If `stores.idempotency` is omitted, Arc starts but logs a warning because idempotent write dedupe would be instance-local.

  helmet: { contentSecurityPolicy: { directives: { defaultSrc: ["'self'"] } } },

## Custom AJV Keywords

External query parsers or schema generators may use non-standard JSON Schema keywords (e.g., `x-internal` from MongoKit). By default, Fastify's AJV strict mode rejects unknown keywords. Use `ajv.keywords` to allowlist them:

```typescript
const app = await createApp({
  ajv: { keywords: ['x-internal'] },
});
```

## TypeBox Type Provider

```typescript
const app = await createApp({
  typeProvider: 'typebox',
});
```

**Requires:** `npm install @sinclair/typebox @fastify/type-provider-typebox`

```typescript
import { Type, ArcListResponse, ArcPaginationQuery } from '@classytic/arc/schemas';

const ProductSchema = Type.Object({ _id: Type.String(), name: Type.String() });

fastify.get('/products', {
  schema: { querystring: ArcPaginationQuery(), response: { 200: ArcListResponse(ProductSchema) } },
}, handler);
```

| Helper | Description |
|--------|-------------|
| `ArcListResponse(schema)` | Paginated list — `oneOf` of `{ method, data, ... }` (offset / keyset / aggregate) or bare `{ data }` |
| `ArcOffsetListResponse(schema)` | Pin one variant — `{ method: 'offset', data, page, limit, total, pages, hasNext, hasPrev }` |
| `ArcKeysetListResponse(schema)` | Pin one variant — `{ method: 'keyset', data, limit, hasMore, next }` |
| `ArcAggregateListResponse(schema)` | Pin one variant — `{ method: 'aggregate', ...same as offset }` |
| `ArcBareListResponse(schema)` | Pin one variant — `{ data }` |
| `ArcDeleteResponse()` | Delete — `{ message, id?, soft? }` |
| `ArcErrorResponse()` | Error — canonical `ErrorContract` from `@classytic/repo-core/errors` |
| `ArcErrorDetail()` | Single field-scoped error detail — embed in custom 422 / 409 schemas |
| `ArcPaginationQuery()` | Query params — `{ page?, limit?, sort?, select?: string \| string[] \| Record<string, 0\|1> }` |

Single-doc responses (`get` / `create` / `update`) intentionally have no helper — the doc IS the response. Pass your TypeBox schema directly to `response: { 200: schema }`.
